网络请求后的数据统一处理

Observable内部处理(但是处理仍然处于子线程中)

  • 在Observable.subscribe(observer/subscriber)之前，此时网络请求已经完成，但是还是在子线程中并没有下发给观察者(observer/subscriber),所以这个时候可以将Observable得到的数据进行转换拦截，当遇到合理的情况的时候在下发给观察者，当不合理时进行拦截，例如Token失效跳转登录等

  • Observable的数据处理转换并且处理后还需要返回个Observable，因为外部需要进行订阅观察者，所以其实相当于之前的Observable只是做了网络请求，当请求完成后真正跟UI层传递的的Observable其实是在之前的那个Observable中转换后生成的，也可以理解为是数据做过处理后创建的(因为数据请求后本身Retrofit返回的是Observable的对象，所以数据想处理后返回Obeservable,其实是不可能的，所以就用到其内部的flatMap、Map方法将其Observable的内部数据进行转化，在其转换背部的Call方法中在重新创建个新的Observable对象传递给外部的UI去订阅Observer)

private <T extends HttpResponse> Observable<T> wrapper(Observable<T> resourceObservable) {
return resourceObservable
        .doOnSubscribe(new Action0() {
            @Override
            public void call() {
            }
        })
        .flatMap(new Func1<T, Observable<T>>() {//Func方法可以理解为通过处理T返回Observable<T>,
            @Override
            public Observable<T> call(final T baseResponse) {
                Observable<T> observableB = Observable.create(new Observable.OnSubscribe<T>() {
                    @Override
                    public void call(Subscriber<? super T> subscriber) {
                        if (baseResponse == null) {
                            subscriber.onCompleted();
                        } else {
                            //success
                            switch (baseResponse.code) {
                                case HttpConfig.TOKEN_PAST:

                                    break;
                                default:
                                    subscriber.onNext(baseResponse);
                                    break;
                            }
                        }
                    }
                });
                return observableB;//返回新创建的Observable提供给外部
            }
        })
        .doOnError(new Action1<Throwable>() {
            @Override
            public void call(Throwable throwable) {
                //网络异常超时处理
            }
        });
}

Subscriber(Observer)内部处理

  • 这种方式是不在Observable哪一步拦截，而是让其原先的Observable进行派发，然后在订阅的时候拦截，也就是Subscriber哪一步拦截，但是这一步已经是UI线程了，如果做统一的处理后在各个给不同的页面组件处理，就需要在定义一个回调接口，然后在统一的处理中进行业务判断，然后在调用回调接口，传递给各个页面进行不同的更新UI
  • 一般是自定义Subscriber类实现内部的onNext，onError，onComplete方法，然后在不同方法进行业务拦截判断，在调用不同的success、fail、error方法进行更新Ui

  • 这种方式相对来说好处理一点，因为其已经在UI线程只了所以其弹Toast很方便不用在创建线程判断线程，考虑Looper问题，而且你还可以进行Dialog的统一控制显示隐藏，因为在进行订阅时subscriable.subscribe(subscriber)，其实就是网络请求的启动，也就是Subscriber的创建过程，所以可以在Subscriber的onStart方法中进行Dialog的显示，并在onNext/onError中进行隐藏等，所以其相对来说方便一点。
